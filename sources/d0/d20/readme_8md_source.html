28 ## Base Definitions
29 
30 ### <a name="belief">Beliefs</a>
31 
32 * Beliefs implicitly describe the current state of the agent
33 * Beliefs will be updated before the cycle is run (beliefbase uses an update mechanism)
34 * Beliefs must be exists iif an expression is computed (beliefs can be exist on the fly)
35 * Belief addition triggers a plan with the definition ```+belief```
36 * Belief retraction triggers a plan with the definition ```-belief```
37 * Belief modification with ```-+``` does not exists anymore
38 * [Variables](#variable) within a belief literal will be unified before the belief is added to the beliefbase
39 
40 
41 ### <a name="action">Actions</a>
42 
43 * Actions will be run immediately
44 * Actions can fail (false) or succeed (true)
45 * There is no difference between internal and external actions
46 * Actions can be also a logical or assignment expression (these are always true)
47 * Actions with ```@```-prefix wil be executed in parallel (each inner action will be run in parallel)
48 
49 
50 ### <a name="plan">Plans</a>
51 
52 * Plans are _sequences of [actions](#action), [rules](#rule) and/or achievement / test [goals](#goal)_
53 * Plans has got an optional context, that defines a constraint for execution (default is true and matches always)
54 * Plans fail iif an item of the plan fails
55 * Plans returns a boolean value which defines fail (false) and success (true)
56 * Plans run items in sequential order on default
57 * If the plan calls an _achievement [goal](#goal) addition_, the [goal](#goal) is added to the global [goal](#goal) list and the current plan will be continued, the _achievement [goal](#goal)_ will be run within the next cycle
58 * An _achievement [goal](#goal) deletion_ does not exists anymore
59 * All items results will be concatenated with a logical _and_ to calculate the plan result value
60 
61 #### <a name="planinternal">Internals Constants</a>
62 
63 * The plan has got additional [constant variables](#variable), that are added in the context condition (values are calculated before plan execution is started)
64  * _Score_ returns the current score-value of the plan
65  * _PlanFail_ stores the number of fail runs and _PlanFailRatio_ normalized value in [0,1]
66  * _PlanSuccessful_ stores the number of successful runs and _PlanSuccessfulRatio_ normalized value in [0,1]
67  * _PlanRuns_ number of runs of the plan (fail + successful runs)
68  * _Cycle_ agent-cycle number
69 
70 #### <a name="fuzzy">Fuzziness</a>
71 
72 * Fuzzy value must be in [0,1]
73 * Each [action](#action) in a fuzzy-plan returns also a fuzzy value to define the fuzziness
74 * The [plan](#plan) or [rule](#rule) result returns true / false and the aggregated fuzzy value
75 
76 
77 ### <a name="rule">Rules</a>
78 
79 * Rules are similar to [plans](#plan) without the context condition
80 * Rules cannot be triggered by a goal, so they must be called from a plan
81 * Rules will be executed with the prefix ```$```
82 * Rules run immediatly
83 * Rules run sequentially on default
84 * Rules returns a boolean value which defines fail (false) and success (true)
85 * All items results will be concatinate with a logical _and_ to calculate the plan result value
86 * [Variables](#variable) will be passed, so if a rules succeed the value of the variable will be passed back to the calling plan
87 
88 
89 ### <a name="annotation">Rule / Plan Annotation</a>
90 
91 * Annotations can modify a plan / rule behaviour to change runtime semantic
92 * The following annotation can be used
93  * ```@Score``` adds an individual score value
94  * ```@Atomic``` the plan / rule cannot be fail, it returns always true (only the [actions](#action) can fail)
95  * ```@Parallel``` all items will be run in parallel
96 
97 ### <a name="goal">Goals</a>
98 
99 * Semantically a goal marks a certain state of the world an agent _wishes to bring about_ [AgentSpeak, p.40]
100 * _Achievement goals_ triggers an _achievement goal addition_ which leads to the execution of a corresponding [plan](#plan)
101 * On agent start, there can exists one _initial goal_ only (like the ```main``` function in Java, C/C++)
102 * Each agent can track _more than one goal_ at the same time otherwise the agent idles (the suspending state is not used)
103 * Goals are triggered by external events which will match by the goal name
104 * Goals will be resolved into [plans](#plan) with equal name (and allowed context), the [plan](#plan) is the intantiiation of the goal
105 * Goals are run in parallel independed from other goals
106 * A goal is a sequence of [plans](#plan) which must all finished successfully
107 * A goal is part of exactly one [intention](#intention)
108 * If a goal can match a [desire](#desire) (the goal is near to the desire) it can add an event to match the desire [belief](#belief)
109 * If the agent is in sleeping / hibernate state and the ```wakeup``` method is called, it triggers the wakeup-goal
110 
111 #### <a name="testgoal">Test Goals</a>
112 
113 * A test goal is an atom with the definition ```?literal```
114 * The test return true iif a plan with an equal literal is within the current execution context (current running)
115 
116 
117 ### <a name="intention">Intentions</a>
118 
119 * An intention is the _convex hull_ of its [goals](#goal)
120 * The intention is set of of goals, which must exist simultaneously
121 * Intentions cannot be in conflict with other intentions, so there dies not exists any overlaping
122 
123 
124 ### <a name="desire">Desires</a>
125 
126 * A Desire is a vertex of the edge of all intentions
127 * Desires are defined by a set of beliefs
128 * Desires can be in conflict with other desires, represented that the desires have got a large distance (much as possible)
129 * The desire is successfully reached, iif all beliefs are existing anytime

131 
132 ### <a name="variable">Variables</a>
133 
134 * Variables are written with an upper-case letter at begin
135 * Thread-safe variables for parallel runtime start with ```@``` (at-sign) followed by an upper-case letter
136 * Variables can store a literal or string to call a [rule](#rule) or [plan](#plan) e.g. ```!X(3,2)``` calls a plan or ```$X(2,1)``` calls a rule
137 
138 ### <a name="at-annotation">Action / Term Annotation</a>
139 
140 * In LightJason one can specify HOW actions and terms will be executed / unified.
141 * Concept of ```action-term-annotation```s allows to annotate ```actions```, and ```terms``` to perform
142  * unification (```>>```)
143  * parallel execution (```@```), see [Variables](#variable) and _lambda expressions_.
144  * ...
145 * If more than one ```action-term-annotation``` is needs to be added, they have to be ordered according to the rule: _First HOW, then WHAT_, e.g. ```@>>``` (parallel unification)
146 * To annotate multiple actions/terms brackets ```(```,```)``` can be used. See the following examples
147 * Examples
148  * ```@>>( foo(X), X > 1 ) && Score > 0.5``` (unify ```foo(X)``` and ```X > 1``` in parallel and if this results in a true statement check whether ```Score > 0.5```)
149  * ```>>foo(X) && X > 1 && Score > 0.5``` (unify ```foo(X)```, then test the following terms sequentially)

164 ## <a name="cycle">Running Semantics - Agent-Cycle</a>
165 
166 Semantik definition of Jason see chapter 10.1 [AgentSpeak, p.207]
167 
168 1. execute perceiving on each beliefbase
169  * run _perceive_ on each beliefbase perceiving object
170  * add / delete beliefs in the beliefbase
171  * beliefbase events will generate
172 
173 2. if agent is in suspend state stop execution
174 
175 3. run agent cycle
176 
177  1. run update of defuzzifcation
178  2. collect plans, which match belief-events and goal-events
179  3. create plan execution list of instantiated plans
180  4. execute instantiated plans in parallel and apply the following rules for each body formula
181 
182  1. if an item is an _action_ or _rule_ execute it immediatly
183  2. if an item is a _test goal_ check instantiated plans if a plan is found return true otherwise false
184  3. if an item is an _achievment goal_ and
185  * begins with ```!``` add a new trigger-event for the next cycle to instantiate a possible plan
186  * begins with ```!!``` the plan which is matched by the goal is executated immediately within the current plan context
187  4. if an item is a belief-
188  * addition, unify the literal and add it into the beliefbase and create a trigger-event
189  * deletion, unify the literal and remove it from the beliefbase and create a trigger-event
190 
191  5. if a plan is finished check plan result, if it is false / fail create a ```-!goal``` event
192 
193 4. increment cycle value
